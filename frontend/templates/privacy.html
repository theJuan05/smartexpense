2 Threat Modeling & Defenses

We identified the following major threats and implemented specific countermeasures for each.

SQL Injection
An attacker submits malicious SQL through form inputs to read or destroy the database.
✅ Defense: All queries use parameterized statements via PyMySQL — user input is never concatenated into SQL.
Man-in-the-Middle Attack
An attacker intercepts network traffic between your device and the server to read or modify data.
✅ Defense: All connections are encrypted with HTTPS/TLS. HTTP requests are rejected.
Database Breach
An attacker gains direct access to the database and reads stored financial data.
✅ Defense: Expense titles and notes are encrypted with AES-256-GCM. Even with raw DB access, data is unreadable without the server-side key.

3 Encryption Details

Data at Rest: Sensitive expense fields (title, notes) are encrypted using AES-256-GCM via Python's cryptography library. A 256-bit key is derived from a server-side secret using SHA-256. Each value is encrypted with a unique 96-bit random nonce, making ciphertexts non-deterministic.

Data in Transit: All API communication is secured with TLS 1.2+ (HTTPS). The app is deployed on AWS EC2 with Let's Encrypt SSL, enforcing HTTPS on all endpoints.

Passwords: User passwords are never stored in plaintext. They are hashed using a secure one-way algorithm (bcrypt/PBKDF2) before storage.

Authentication: Sessions are protected with signed JWT tokens (algorithm: HS256). Tokens carry an expiry and are verified on every authenticated request.

4 Your Rights & Data Control

You have full control over your data at all times:

• Access: All your expense data is visible in the app at any time.

• Delete: You can delete individual expenses or clear all your data from the Settings page.

• Account deletion: Deleting your account permanently removes all associated data from our servers.

• No third-party sharing: Your financial data is never sold, shared, or disclosed to any third party.
